1. Overview

커리업이 무엇이고, 누구를 위해, 왜 만들었는지.

한 줄 정의

커리업은 취업 준비생이 지원 회사·면접 일정·이력서/자소서·AI 피드백을 한 곳에서 관리하는 웹 대시보드다.

왜 만들었나

  • 취준생은 보통 노션·엑셀·캘린더·메모 앱에 흩어진 정보로 지원 현황을 추적함 — 마감을 놓치고, 회사가 헷갈리고, 면접 일정이 충돌
  • "내가 어디까지 진행했는지" 한눈에 보여주는 단일 진실원(single source of truth)이 필요했음
  • 자소서·면접 준비도 AI가 도와주면 시간 단축

누구를 위해

  • 현재 동시에 5개 이상 회사에 지원하고 있는 취준생
  • 이직 준비 중인 직장인
  • 인턴·공채·수시 채용을 함께 관리하는 학생

4. 분기 처리

유저 상태별 화면·라우팅 분기. 미들웨어와 컴포넌트 단계 모두 포함.

4.1 미들웨어 라우팅 (proxy.ts)

Next.js 16에서 middleware.ts → proxy.ts로 변경. 모든 요청에서 4단계 검사 실행.

단계 조건 결과
1. 미인증 user 없음 + 비공개 경로 → /login redirect (jt_onb 쿠키 정리)
2. 로그인 페이지 우회 user 있음 + /login 진입 → / redirect
3. 온보딩 게이트 쿠키 없음 + DB onboarded_at NULL → /onboarding redirect
4. 온보딩 완료자 진입 완료한 유저가 /onboarding 진입 → / redirect

예외 (proxy 통과): /api/*, 정적 자산(_next/*, *.svg, *.png, sitemap.xml, robots.txt, manifest.webmanifest) — manifest는 redirect되면 PWA 아이콘이 fallback돼서 명시적 제외.

4.2 사이드바 노출

아래 경로에서는 사이드바 숨김 (LayoutWrapper):

  • /login, /forgot-password, /reset-password — 인증 페이지
  • /onboarding — 집중 흐름
  • /admin — 별도 레이아웃 (관리자 헤더)
  • /terms, /privacy — 약관 (외부 deep link 접근)

4.3 ApplicationStatus 분기

4.4 회원가입 화면 내부 분기

단일 페이지에서 모드(login/signup)·결과 상태에 따라 4가지 뷰가 나옴:

  • 로그인 폼 — 기본
  • 회원가입 폼 — 가입 절차 안내 + 비밀번호 강도 + honeypot
  • "메일함 확인" 화면 — 가입 후 폼이 사라지고 등장. 입력한 이메일을 굵게 표시
  • "비밀번호 재설정 완료" 배너 — /login?reset=ok 진입 시

4.5 비밀번호 정책 분기

실시간으로 5개 규칙 통과 여부 체크. 실패 시 한국어 에러:

{ id: "length",  test: p.length >= 8 && p.length <= 16 }
{ id: "lower",   test: /[a-z]/.test(p) }
{ id: "upper",   test: /[A-Z]/.test(p) }
{ id: "digit",   test: /[0-9]/.test(p) }
{ id: "special", test: /[^A-Za-z0-9]/.test(p) }

RLS 정책

모든 사용자 데이터 테이블은 동일한 패턴:

ALTER TABLE companies ENABLE ROW LEVEL SECURITY;
CREATE POLICY "users can manage own companies"
  ON companies FOR ALL USING (auth.uid() = user_id);

application_events처럼 간접 소유는 EXISTS 서브쿼리로 검증.

인덱스 · 트리거

  • job_applications.status, calendar_events.starts_at도 인덱스
  • updated_at은 update_updated_at() 트리거로 자동 갱신
